# 6. 核心业务逻辑说明（重点）

## 6.1 模块1：站点数据查询（ChargePadLineController）

### 功能作用
提供生产线站点列表查询和站点数据记录查询，是前端看板获取展示数据的主要入口。

### 调用入口
- GET `/api/ChargePadLine` - 获取所有站点列表
- POST `/api/ChargePadLine` - 根据站点获取最新数据记录

### Service层逻辑（ChargePadLineService）

```csharp
// 获取站点列表
public async Task<List<StationList>> GetStationListAsync()
{
    return await _stationListRepo.GetListAsync();
}

// 根据站点获取最新数据
public async Task<DataList?> GetDataListByStationAsync(DataSearchDto searchDto)
{
    var stationId = searchDto?.StationId;
    return await _dbContext.DataLists
        .Where(x => x.StationListId == stationId)
        .OrderByDescending(x => x.CreateTime)
        .FirstOrDefaultAsync();
}
```

### Repository访问方式
- 使用通用 `Repository<TEntity>` 进行数据访问
- `GetListAsync()` 获取所有站点
- EF Core 直接查询获取最新数据记录

### 数据处理流程
```
前端请求 → Controller验证参数 → Service查询 → 按CreateTime倒序取最新一条 → 响应JSON
```

### 异常处理机制
- 参数验证：检查 StationId 是否为空
- 异常捕获：try-catch 包裹，返回统一错误格式

## 6.2 模块2：MES数据接收（LatestDataController）

### 功能作用
接收MES系统推送的站点设备测试数据，支持新增和更新（Upsert）逻辑。

### 调用入口
- POST `/api/LatestData` - 接收MES推送数据

### Service层逻辑（LatestDataService）

```csharp
public async Task<bool> UpsertLatestDataAsync(LatestDataRequestDto request)
{
    // 1. 查找对应站点
    var station = await _dbContext.StationLists
        .FirstOrDefaultAsync(s => s.StationId == request.StationId);

    // 2. 如站点不存在，记录警告并返回
    if (station == null)
    {
        _logger.LogWarning("未找到对应站点...");
        return false;
    }

    // 3. 查询该站点已有数据
    var existingData = await _dbContext.DataLists
        .Where(x => x.StationListId == station.StationId)
        .OrderByDescending(x => x.CreateTime)
        .FirstOrDefaultAsync();

    // 4. 执行 Upsert：新增或更新
    if (existingData == null)
    {
        // 新增
        var newData = new DataList { ... };
        _dbContext.DataLists.Add(newData);
    }
    else
    {
        // 更新
        existingData.SNnumber = request.SNnumber;
        existingData.Data = request.Data;
        existingData.CreateTime = DateTimeOffset.UtcNow;
        _dbContext.DataLists.Update(existingData);
    }

    await _dbContext.SaveChangesAsync();
    return true;
}
```

### 数据处理流程
```
MES推送 → 验证StationId和Data → 查找站点 → 查询现有数据 → Upsert → 保存 → 返回结果
```

### 异常处理机制
- 参数验证：检查 StationId 非空、Data 非空
- 异常捕获：try-catch 包裹，LogError 记录，返回 false

**数据更新方式**：完全依赖MES系统主动推送，实时性由MES系统控制。

**如需新增定时任务**，可参考以下方式：

```csharp
// 在 Program.cs 中注册后台服务
builder.Services.AddHostedService<MyBackgroundService>();

// 实现 BackgroundService
public class MyBackgroundService : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            // 执行定时任务逻辑
            await Task.Delay(TimeSpan.FromMinutes(5), stoppingToken);
        }
    }
}
```

# 10. 部署说明（详细）

### 方式1：直接部署

```powershell
# 发布 Release 版本
dotnet publish -c Release -o ./publish
```

### 方式2：发布到文件夹

```powershell
dotnet publish -c Release -o d:\deploy\X509Data
```

## 10.2 发布后目录结构

```
d:\deploy\X509Data\
├── ChargePadLine.Api.dll          # 主程序集
├── ChargePadLine.Api.exe          # 可执行文件
├── appsettings.json               # 配置文件（需根据环境修改）
├── log4net.config                # 日志配置
├── logs/                          # 日志输出目录
└── [...其他依赖文件]
```

## 10.3 IIS 部署配置

1. 安装 ASP.NET Core Hosting Bundle
2. 创建 IIS 网站，指向发布目录
3. 配置应用程序池：`No Managed Code`，`Integrated` 模式
4. 配置 `web.config`（如需）：

```xml
<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <location path="." inheritInChildApplications="false">
    <system.webServer>
      <handlers>
        <add name="aspNetCore" path="*" verb="*" modules="AspNetCoreModuleV2" resourceType="Unspecified"/>
      </handlers>
      <aspNetCore processPath="dotnet" arguments=".\ChargePadLine.Api.exe" stdoutLogEnabled="false" hostingModel="inprocess"/>
    </system.webServer>
  </location>
</configuration>
```

## 10.4 配置文件替换

发布后需根据环境修改 `appsettings.json`：

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",

  "Urls": "http://0.0.0.0:5000",
```

通过修改 `appsettings.json` 中的 `ConnectionStrings.AppDbContext` 实现：

- 生产环境：连接生产服务器 PostgreSQL

# 11. 常见问题排查（非常详细）

## 问题1：数据库连接失败

**问题现象**：
```
NpgsqlException: Failed to connect to host=localhost port=5432
```

**可能原因**：
- PostgreSQL 服务未启动
- 连接字符串错误（Host、Port、Database）
- 用户名/密码错误
- 防火墙阻止连接

**排查方式**：
1. 检查 PostgreSQL 服务状态：`services.msc` 中确认 postgresql-x64-16 运行中
2. 测试连接：`psql -h localhost -U postgres -d X509Data`
3. 检查 `appsettings.json` 连接字符串

**解决方案**：
- 启动 PostgreSQL 服务
- 修正连接字符串
- 检查用户名密码

# 12. 日志位置说明

## 12.1 日志目录

| 环境 | 日志目录 | 说明 |
|------|----------|------|
| 本地开发 | `d:\code\X509设备参数看板后端\X509Data\logs\` | 相对于应用程序运行目录 |
| IIS部署 | `{部署目录}\logs\` | 需创建并赋予写入权限 |

## 12.2 日志级别

| 组件 | 日志级别 | 配置位置 |
|------|----------|----------|
| 全局 | INFO | log4net.config root level |
| Modbus.Service.Cache | ERROR | log4net.config logger |

## 12.3 日志文件命名规则

```
logs/yyyyMMdd.txt
例如：logs/20260518.txt
```

## 12.4 日志格式

```
2026-05-18 10:30:00 [线程ID] INFO  命名空间.类名 - 日志消息
```

## 12.5 如何定位错误

1. **查看日志文件**：在 logs 目录下找到对应日期的日志文件
2. **搜索 ERROR 关键字**：定位异常信息
3. **根据异常堆栈**：找到出错的代码位置

## 12.6 如何开启 Debug 日志

修改 `log4net.config`：

```xml
<root>
  <level value="DEBUG" />  <!-- 改为 DEBUG -->
  <appender-ref ref="ConsoleAppender" />
  <appender-ref ref="RollingLogFileAppender" />
</root>
```

# 13. 已知技术债务 / 风险点（必须诚实）

## 13.1 设计不合理之处

| 问题 | 影响 | 建议 |
|------|------|------|
| CORS 允许任何来源 | 存在安全风险，仅适合内网使用 | 生产环境应配置具体允许的来源 |
| 认证/授权为占位状态 | 接口无访问控制 | 建议实现 JWT Token 认证 |
| 无数据清理机制 | 长期运行数据表会持续膨胀 | 建议添加定时清理任务 |

## 13.2 临时方案

| 问题 | 说明 |
|------|------|
| 数据库初始化使用 EnsureCreated | 不会执行 Migration，生产环境建议使用 Migration |
| 无数据库迁移脚本 | 建议添加正式的数据库版本管理 |

## 13.3 容易出问题的模块

| 模块 | 风险点 |
|------|--------|
| LatestDataService.UpsertLatestDataAsync | 并发推送时可能存在竞态条件 |
| Repository | GetListAsync 重载过多，代码冗余 |

## 13.4 需要重构的地方

| 重构项 | 说明 |
|--------|------|
| 统一日志框架 | log4net 配置较分散，建议统一结构化日志 |
| 添加数据验证层 | 建议使用 FluentValidation 替代手动验证 |
| 添加单元测试 | 当前无测试覆盖 |

## 13.5 表结构风险

| 表名 | 风险 |
|------|------|
| X509Data | 无分区设计，数据量大时查询性能会下降 |
| X509Data.Data | 存储格式为字符串拼接（电压:220V\|电流:5A），不便于统计查询 |
